## 1. 設計コンセプト

タイムレコード利用開始に必要な4項目（業態・ブランド／店舗／雇用区分／従業員）の登録について、**「新規登録モード」と「運用登録モード」を画面・UIフィードバック・順序強制度で完全分離** します。

### 設計原則

| 観点 | 新規登録モード | 運用登録モード |
|---|---|---|
| **目的** | 利用開始のための4項目必須登録 | 既存運用での追加・編集 |
| **順序** | 必須（① → ② → ③ → ④） | 任意 |
| **強制度** | 強誘導（CTAモーダル） | 軽量示唆（CTAトースト） |
| **画面** | `setup-*` 専用ページ（ウィザード式） | `*-settings` 通常設定画面（一覧+追加） |
| **UI** | クリーンなステップカード | 一覧表 + 「+ 追加」 |

## 3. 画面仕様

### 3.1 B-0 登録ガイド (`#b0`)

**役割**: 4項目登録の進捗可視化・順序提示・ステップ画面への誘導ハブ

**最上部の構成（共通）**:
- `b0-completed-banner`: done=4 のときに「🎉 初期登録は完了しています」を表示
- 登録方法選択カード `method-picker`: 「設定画面で1件ずつ登録」/「CSVで一括インポート（推奨）」を選択。初期表示は **CSV一括（推奨）**。

#### 3.1.1 manualモード（設定画面で1件ずつ登録）

`ow-card` 内に **4ステップカード（ow-step）** のみを縦並びで表示:
- 1: 業態・ブランド登録 / 2: 店舗登録 / 3: 雇用区分登録 / 4: 従業員登録
- 各カード: ステップ番号 + タイトル + 説明 + 前提登録 + 「登録済み: N件」 + CTAボタン
- CTAボタン: 「登録する →」（未完了） / 「再編集 →」（完了済み）
- ロック表示: 前ステップ未完了時は `is-locked` + ボタン disabled
- ❗ 進捗リング・パーセント・残数・ステップマーカーは **非表示**（2026-05-11 簡素化）

#### 3.1.2 importモード（CSV一括インポート）（推奨）

`import-panel` 内に以下を縦並びで表示:

1. **方式選択（分岐枠）**: 独立ボックスでピル選択
   - 📋 テンプレートで取り込み（推奨タグ）
   - 🔄 外部勤怠サービスから取り込み
2. **📋 インポートガイド** リスト:
   - テンプレートモード時: **Step 1〜4**（業態・ブランド / 店舗 / 雇用区分 / 従業員）
   - 外部連携モード時: **Step 1〜2**（Step 1=業態・ブランド(テンプレ補完) / Step 2=店舗・雇用区分・従業員(外部一括)）
   - 各行: 番号アイコン + タイトル + 状態バッジ（進行中/✓完了/🔒 Step N完了後）+ CTA
3. **📌 後追加項目（推奨）**: 管理者をテンプレで取り込み（任意・モード共通）
4. **🛠 デモ用シミュレーション**: 静的モック⇄動的モックの連携擬似ボタン

> ⚠️ **B案採用（2026-05-11）**: 「方式選択」は分岐選択（性質: ピル即時切替）でガイドリストとは別枠。ガイドリストは **実取込作業のみ**（性質: 画面遷移＋数分かかる）を Step 1 から番号付け。

**CTAボタン共通仕様**:
- 「インポート画面を開く →」 → setup-import-* に遷移
- 「外部連携画面を開く ↗」 → 別タブで rakmy-timerecord-import-spec/#saas-setup
- 「管理者をテンプレで取込 ↗」 → 別タブで rakmy-timerecord-import-spec/#tpl-setup
- 順序違反は `is-locked` でガード

**完了後（done=4）**:
- 最上部の `b0-completed-banner` のみで通知（インポートパネル内・ウィザード内の重複バナーは廃止）
- サイドバーから「📋 登録ガイド」項目は非表示

### 3.2 setup-* 新規登録ステップ画面（4種）

各画面共通レイアウト:
- **サイドバー**: 「📋 登録ガイド」+ 「初期登録ステップ」(①②③④進捗マーク付き)
- **メインエリア**:
  - 「← 登録ガイドへ戻る」リンク
  - 中央に **緑枠ステップカード**: STEP X/4 バッジ + タイトル + 説明 + 前提登録/件数情報 + 「+ 登録する」大型ボタン

**動作**:
- 「+ 登録する」 → 該当登録モーダル展開
- モーダル登録完了 → CTAモーダル（橋渡し）→ 次の setup-* へ

### 3.3 *-settings 運用登録画面（4種）

各画面共通レイアウト:
- **サイドバー**: タイムレコ通常メニュー（打刻記録/勤務時間集計/給与/設定セクション）
- **メインエリア**:
  - タイトル + 「+ 追加」ボタン
  - 一覧表（既登録レコードを表示）

**動作**:
- 「+ 追加」 → 該当登録モーダル展開
- モーダル登録完了 → トーストCTA（右上）→ 任意で次の登録に連鎖

### 3.4 b-step-modal-* 登録モーダル（4種）

各エンティティ専用の純粋な登録フォーム。新規/運用 共通で利用。

| モーダル | フィールド |
|---|---|
| ① 業態・ブランド | 業態select / ブランド名 / ブランドコード |
| ② 店舗 | 店舗名 / 所属ブランド select / 住所 / 営業時間 |
| ③ 雇用区分 | 区分名 / 種別 / 時給単位 |
| ④ 従業員 | 名前 / 電話 / メール / 雇用区分select / 所属店舗select |

**仕様**:
- 純粋な登録フォームのみ（前提充足チェック警告は表示しない）
- 「キャンセル」「✕」 → モーダルクローズ
- 「登録」 → 進捗保存 + モーダルクローズ + モード別CTA表示

### 3.5 CTAモーダル（橋渡しモーダル）

**新規登録モード時のみ表示**。中央オーバーレイで強誘導。

```

新しい◯◯を登録しました。
```

| 登録項目 | メッセージ | 主CTA | 副CTA |

| ① ブランド | 続けて、このブランドの店舗も登録しますか？ | 店舗の登録に進む → | 登録ガイドへ戻る |

| ③ 雇用区分 | 続けて、この区分の従業員も登録しますか？ | 従業員の登録に進む → | 登録ガイドへ戻る |

### 3.6 CTAトースト

**運用登録モード時のみ表示**。画面右上スライドイン。


**動作**:
- 8秒で自動消去
- マウスホバー → タイマー停止

```js
const isInitialMode = b0LoadDone() < 4;
const isOperationalMode = b0LoadDone() >= 4;
```

### 5.3 順序ガード

- `setup-N` が `done + 1` 以外なら警告バナー表示 + ボタン disabled
- `setup-N` で N <= done なら「再編集」モードで動作
- `b0CompleteStep(N)` 内で N !== done+1 の登録は alert で阻止

## 7. 実装にあたり確認事項（エンジニア向け）

### 7.1 ルーティング

- `/setup/brand`, `/setup/store`, `/setup/emptype`, `/setup/emp` （新規登録モード）
- `/settings/brand`, `/settings/stores`, `/settings/emptypes`, `/settings/employees` （運用モード）
- 初回ログイン時は `/setup/brand` から自動開始
- 全項目登録完了後は `/setup/*` URLは無効化（→ `/dashboard` リダイレクト）

### 7.2 モード判定ロジック

- バックエンド側で「初期登録完了フラグ」を保持するか、各テーブルの件数で判定するか
- 推奨: `companies.onboarding_completed_at` カラムを追加し、4項目すべてに1件以上の時刻を保存

### 7.3 順序ガード

- フロント側ガード: 順序違反URLにアクセス → 正しいステップへリダイレクト
- バックエンド側ガード: 必要な前提が未登録なら API側で 422 エラー返却

### 7.4 CTAモーダルの中断UX

- 「登録ガイドへ戻る」を選んだ場合、`b0` 画面で進捗バー＋次ステップ強調で再開できる
- 中断状態を sessionStorage（or DBに）に保存

### 7.5 トーストの実装

- アニメーション: スライドイン（ease 350ms）
- ホバー停止: `mouseenter` でタイマークリア、`mouseleave` で 3000ms タイマー再設定
- アクセシビリティ: `role="status"` `aria-live="polite"` 推奨

### 7.6 招待メール送信（従業員登録）

- 登録時に自動送信ではなく、**従業員一覧の「招待送信」ボタンから手動送信** が確定仕様
- 招待メールテンプレート: 既存 C-4 を「招待送信ボタン経由」に修正済み

## 9. インポート機能との連携（重要）

### 9.1 2つのインポート導線

新規登録ガイドで「CSV一括インポート」を選択した場合（方式選択は分岐枠、Step番号外）、および運用後の再更新時、別リポジトリの **タイムレコード インポート機能** へ別タブで遷移する。

#### 導線1: 新規登録ガイドから（ガイド順インポート推奨）
- 起点: b0「CSV一括インポート」カード → 方式選択（テンプレ/外部）→ ガイドリスト Step 1 の「インポート画面を開く →」
- 推奨フロー（テンプレ）: ガイド順（Step 1 業態・ブランド → Step 2 店舗 → Step 3 雇用区分 → Step 4 従業員）
- 推奨フロー（外部）: Step 1 業態・ブランド(テンプレ補完) → Step 2 店舗・雇用区分・従業員(外部一括)
- 戻り先: 元の b0 タブで進捗確認

#### 導線2: 各設定画面から（運用時再更新）
- 起点: 各設定画面（brand-settings 等）ヘッダーの「📥 CSVインポート ↗」
- 用途: 任意項目の更新・追加（例: 新店舗10件を一括追加）
- 戻り先: 元の設定画面タブ

### 9.2 連携先

- リポジトリ: `suguru789987/rakmy-timerecord-import-spec`
- 公開モック: https://suguru789987.github.io/rakmy-timerecord-import-spec/
- 機能: テンプレート取り込み / 外部勤怠サービスから取り込み（KoT・ジョブカン）

### 9.3 ⚠️ 動的・静的モック連携の課題

| | 新規登録ガイド | インポート機能 |
|---|---|---|
| モック性質 | **静的**（画面遷移のクリック動作のみ） | **動的**（CSV取込・データ反映） |
| 状態管理 | sessionStorage（モック内のフェイク） | 取込シミュレーション |
| リポジトリ | rakmy-owner-registration-mock | rakmy-timerecord-import-spec |

**課題**: 別リポジトリの sessionStorage は共有されないため、インポートモックでの取込完了が新規登録ガイドの進捗に自動反映されない。

### 9.4 本番実装で必要な連携要件

#### A. インポート完了通知
インポートジョブ完了時、バックエンドから登録ガイド側へイベント通知:
- 推奨方式: Webhook / WebSocket / Server-Sent Events / ポーリング
- 通知内容: 完了したエンティティタイプ、件数、エラー件数

```
{
  "event": "import.completed",
  "entity": "brand",
  "success_count": 3,
  "error_count": 0,
  "completed_at": "2026-05-11T11:30:00+09:00"
}
```

#### B. 登録ガイドの進捗自動更新
完了通知を受信したら:
- b0 の進捗バー / 件数 / ステップマーカーを自動更新
- 必要な4項目すべて充足したら完了サマリ画面に切替

#### F. URLパラメータによる方式・ステップのスキップ（重要）

インポート画面へは **2つの起点で挙動を分ける**:

##### F-1. 新規登録ガイドからの起点（方式選択スキップ）

b0 の **方式選択（分岐枠）** で方式を選択済みなので、インポート画面では **方式選択画面をスキップして直接該当ステップへ遷移**。

URL: クエリパラメータ付き
```
?method=template&step=brand
?method=template&step=store
?method=template&step=emptype
?method=template&step=emp
?method=external&step=store
?method=external&step=emptype
?method=external&step=emp
```

| パラメータ | 値 | 効果 |
|---|---|---|
| `method` | `template` | テンプレート方式の画面で起動 |
| `method` | `external` | 外部連携方式の画面で起動 |
| `step` | `brand` | 業態・ブランドのアップロード画面 |
| `step` | `store` | 店舗のアップロード画面 |
| `step` | `emptype` | 雇用区分のアップロード画面 |
| `step` | `emp` | 従業員のアップロード画面 |

URL例:
```
/import?method=template&step=brand  → テンプレート > 業態・ブランドアップロード画面に直接遷移
/import?method=external&step=store  → 外部連携 > 店舗マッピング画面に直接遷移
```

##### F-2. 運用設定画面（既登録更新）からの起点（方式選択を表示）

各設定画面（brand-settings 等）の「📥 CSVインポート ↗」からの起点は **方式選択を必ず表示**。

URL: クエリパラメータなし
```
/import
```

**理由**: 既に登録済みデータの更新・追加では、ユーザーが状況に応じてテンプレート / 外部連携を都度選びたい。例えば:
- 既存ブランドの店舗を1件だけ追加 → テンプレート方式が便利
- 別の勤怠サービスから一括移行 → 外部連携方式が便利

##### F-3. 2つの起点の差別化（実装まとめ）

| 起点 | URL | インポート画面の表示 |
|---|---|---|
| **新規登録ガイド (b0)** | `?method=X&step=Y` 付き | 方式選択スキップ → 該当ステップ画面 |
| **運用設定画面** | クエリパラメータなし | 方式選択画面（テンプレ/外部）を表示 |

**理由**: ガイド側で既に「テンプレート / 外部連携」を選択済みなのに、インポート画面でも再選択を要求するのはUX重複。クエリパラメータでスキップさせて、ガイド順にスムーズにインポートできるようにする。一方、運用時は柔軟性を残すためデフォルトの方式選択UIを維持する。

**現状のモック**: 既存インポートモック (rakmy-timerecord-import-spec) はクエリパラメータ未対応のため、現状は b0 起点でも方式選択画面が表示される。本番実装時にクエリパラメータ対応すること。

#### G. 管理者データ インポート対応

管理者は新規登録ガイドの4エンティティ（業態・店舗・雇用区分・従業員）に含まれないが、運用更新時の一括登録ニーズがあるため、別途インポート機能を提供。

##### G-1. 対応範囲

| 取り込み方式 | 管理者インポート | 理由 |
|---|---|---|
| テンプレートで取り込み | ✓ 対応 | ラクミー専用テンプレートで管理者を一括登録 |
| 外部勤怠サービスから取り込み | ❌ 非対応 | 外部勤怠サービス（KoT/ジョブカン等）に「管理者」概念が存在しないため |

##### G-2. テンプレート列定義

ファイル名: `ラクミー_インポートテンプレート_⑤管理者.tsv`

| 列名 | 必須 | 入力形式 | 制約 |
|---|---|---|---|
| 管理者名 | ★ | テキスト | 1行につき1名 |
| メールアドレス | ★ | テキスト | 有効なメールアドレス形式、招待メール送信先 |
| 担当店舗 | ★ | テキスト | ② 店舗の店舗名と一致、複数指定は半角カンマ区切り |
| 権限 | ★ | 選択 | オーナー / 管理者（通常は「管理者」） |

##### G-3. 2つの起点（既存パターンと整合）

| 起点 | UI | リンク先 |
|---|---|---|
| **管理者一覧 (D-1)** ヘッダー | 「📥 CSVインポート ↗」ボタン | インポート画面 `#tpl`（テンプレ画面） |
| **インポート画面 `s-tpl1`** | テンプレ選択肢に「⑤ 管理者」 | （同画面内で選択） |

##### G-4. 依存関係

- 管理者は **店舗（② 店舗テンプレ）が登録済み** であることが前提
- 担当店舗列は既存の店舗名を参照（マッピングで紐付け）
- インポート時は店舗存在チェックを実行

##### G-5. 新規登録ガイド（b0）における取り扱い

管理者は **新規登録ガイドのインポートステップには含めない**:
- 理由: 新規登録時はオーナー1人で開始、管理者は運用開始後に追加するのが自然
- 運用開始後、管理者管理画面（D-1）から CSV インポート可能

#### H. インポート画面の「新規登録モード」と「運用更新モード」の切替（重要）

##### H-1. 設計方針: 同一画面 + URLハッシュ切替

インポート画面は **新規登録モード** と **運用更新モード** で同じ画面を使用しつつ、URLハッシュで切替える。完全別スクリーン化は採用しない（保守コスト過大）。

##### H-2. URLハッシュ仕様

| ハッシュ | モード | バナー表示 | 用途 |
|---|---|---|---|
| `#saas-setup` | 新規登録 | **あり**「📋 新規登録ガイド ・ ↩ 登録ガイドへ戻る」 | 新規登録ガイド b0 から遷移 |
| `#tpl-setup` | 新規登録 | **あり**「📋 新規登録ガイド ・ ↩ 登録ガイドへ戻る」 | 新規登録ガイド b0 から遷移 |
| `#saas` | 運用更新 | **なし** | 運用設定画面 / 直接アクセス |
| `#tpl` | 運用更新 | **なし** | 運用設定画面 / 直接アクセス |
| `#` (なし) | 運用更新 | **なし** | 通常起動（方式選択トップ画面） |

##### H-3. バナー仕様（新規登録モード時のみ表示）

- 配置: `.main` 内のメインコンテンツ先頭、`position: sticky; top: 0`
- 全インポート画面（s-saas1 → s-saas-map-edit → s-saas2 → s-saas-preview → s-saas4 等）でスクロール時も常時表示
- 内容:
  - 左: 📋 アイコン
  - 中央: 「新規登録ガイド ・ Step 2: 外部勤怠サービスからインポート」等の状況表示
  - 右: 「↩ 登録ガイドへ戻る」緑塗ボタン
- ボタン挙動: `window.close()` 試行 → フォールバックで b0 URL に遷移

##### H-4. 起点別リンクの設計

| 起点 | リンク先 | バナー表示 |
|---|---|---|
| b0 「外部連携インポート画面を開く」 | `#saas-setup` | あり |
| b0 「管理者をテンプレで取込（後追加項目）」 | `#tpl-setup` | あり |
| D-1 管理者一覧「📥 CSVインポート」 | `#tpl` | **なし**（運用更新の起点） |
| その他運用設定画面の「📥 CSVインポート」 | `#tpl` または `#saas` | **なし**（運用更新の起点） |

##### H-5. 本番実装で必要な対応

1. **クエリパラメータ + ハッシュ対応**: `applyHashRouting()` のロジックを本番にも実装
2. **バナー非表示時の幅・余白調整**: 運用更新モードではバナー分の余白を取らない
3. **完了画面（s-saas4 等）からの自動リダイレクト（任意）**: 新規登録モードで完了したらb0へ自動戻り（または案内）
4. **アクセシビリティ**: バナーに `role="banner"` / `aria-label="新規登録ガイド経由のインポート"` 推奨
5. **ブラウザバック対応**: `hashchange` イベントで適切に状態同期

##### H-6. なぜ「完全別スクリーン」にしないか

| 案 | 評価 | 採用 |
|---|---|---|
| 同一画面 + URLハッシュ切替（採用） | 保守容易、業界標準、要件達成 | ✅ |
| 完全別スクリーン（s-saas1-setup 等を別div） | コード重複、保守2倍 | ❌ |
| 別ファイル化（import-setup.html） | 保守3倍、URLレベル分離 | ❌ |

業務SaaS（freee、SmartHR、KoT等）も同様に「文脈バナー方式」を採用しており、業界標準パターン。

### 9.5 モックでの擬似連携体験

### 12.6 KoT (King of Time) データソース仕様

KOTのエクスポート機能から取得できる4種類のデータをタイムレコのエンティティにマッピングする。s-saas1 の「KoT キングオブタイム」選択時に以下の5チェック項目を表示。

| # | KoTデータソース | 必須/任意 | 初期チェック | タイムレコ側マッピング先 | 備考 |
|---|---|---|---|---|---|
| 1 | **従業員データ** | **必須** | ✓ ON | 従業員 | 従業員コード/所属コード/姓/名/雇用区分コード/メール/日の契約労働時間/週の契約労働日数 |
| 2 | **従業員一覧** | 任意 | OFF | (給与データ補完) | 給与・契約種別を含む補助データ |
| 3 | **所属グループデータ** | 任意 | OFF | **店舗情報** | 所属コード/所属名/所属グループコード/グループ名 |
| 4 | **単価データ** | 任意 | OFF | **雇用区分** | 所属コード/従業員コード/表示名/基本単価/日給or時給 |
| 5 | **管理者データ** | 任意 | OFF | 管理者 | 管理者コード/表示名 |

#### 設計判断
- **必須は「従業員データ」のみ**: KOTの基幹データを最優先で取り込み
- **店舗・雇用区分・管理者**: ユーザーが必要に応じて選択
- **業態・ブランド**: KOTに対応データなし → 「②取り込みデータ」の業態テンプレで補完
- **マッピング先表示**: 各任意項目に「→ タイムレコの○○にマッピング」を補足表示（従業員データは既存挙動のため補足なし）

#### data-map 属性とマッピングパネル
| データソース | data-map | マッピング設定パネル |
|---|---|---|
| 従業員データ | `map-kot-emp` | (既存) |
| 従業員一覧 | `map-kot-hr` | (既存) |
| 所属グループデータ | `map-kot-group` | (本番実装で要追加) |
| 単価データ | `map-kot-rate` | (本番実装で要追加) |
| 管理者データ | `map-kot-admin` | (本番実装で要追加) |

#### 本番実装での留意点
- KOTの実際のCSVファイル名規則（例: `employee_info_YYYYMMDD.csv` / `rate_YYYYMMDD.csv` 等）の確認が必要
- 各マッピング設定パネルでKoT列 → タイムレコ列の対応を定義
- 「単価データ」は雇用区分カラム（給与体系・金額・時給単位）にマップ
- 「所属グループデータ」は店舗カラム（店舗名・所属ブランド）にマップ
- 「管理者データ」は管理者カラム（名前・メールアドレス・担当店舗・権限）にマップ。なお、KOTから取得できる管理者データには **メールアドレスが含まれない** 可能性が高いので、別途入力フォームまたはマージ処理が必要

### 15.1 設計判断の整理（ユーザー指摘 → 方針決定）

ユーザーから2つの本質的な指摘:

1. 「初期登録ステップ毎の登録画面がタイムレコの設定画面ではない、初期登録ステップ専用の登録画面であるところが違和感がある」

## 16. アカウント衝突回避と権限マージ設計（2026-05-13 追加）

### 16.1 問題提起

複数の登録経路(設定画面登録/CSV一括インポート)が存在し、それぞれが「従業員」「管理者」を独立して登録できる仕様だと、**同一メアドに複数の招待メールが届くコリジョン**が発生する。

具体的なシナリオ:
1. 従業員CSVインポートで A さん登録 → 従業員招待メール送信
2. 管理者CSVインポート(新規)で同じ A さん登録 → 管理者招待メール送信
3. → A さんに**2通の招待メール**が届き、PW設定URLが2つ存在しユーザー混乱

### 16.2 採用する設計モデル

「**アカウント = メアド単位、ロールは累積**」モデルを採用する。

```
[アカウントテーブル]
- account_id (UUID, 主キー)
- email (UNIQUE)         ← この一意キーで衝突回避
- password_hash
- status (招待中 / 有効 / 期限切れ)
- created_at

[ロールテーブル]
- account_id (FK)
- role_type (employee / admin)
- stores (担当店舗、複数可)
- emptype (雇用区分・employeeロールのみ)
- granted_at
```

**主要原則**:
- 1メアド = 1アカウント (UNIQUE 制約で物理的に保証)
- 同一アカウントが「従業員ロール」と「管理者ロール」を**累積**して持てる
- 招待メールは「アカウント単位」で1通のみ、ロールは「メール本文と権限付与ロジック」で表現

### 16.3 ケース別の挙動(4パターン)

| # | 状況 | システム処理 | 送信メール |
|---|---|---|---|
| **1** | 同時インポート時に同一メアドが両CSVに存在(PW未設定) | 1アカウント作成 + 両ロール累積付与 | **統合招待メール 1通**「従業員+管理者として招待」 |
| **2** | 既に従業員アカウントあり → 管理者ロール追加 | 既存アカウントに管理者ロール追加 | **権限拡張通知 1通**(PW設定不要、既存PWで全ロール利用可) |
| **3** | 既に管理者アカウントあり → 従業員ロール追加 | 同上(ロール追加方向のみ異なる) | 同上(権限拡張通知) |
| **4** | 同一CSV内でメアド重複 | バリデーションエラー(インポート拒否) | (送信なし)CSV修正要求 |

### 16.4 設定画面登録での実装

#### 16.4.1 STEP 4 従業員登録モーダル

- 「登録 & 招待送信 (N件)」ボタンで一括登録+招待メール送信
- 内部処理: 各メアドに対して「アカウント新規作成 OR 既存アカウントに従業員ロール追加」をAPIが判定
- 既存管理者アカウントが存在する場合は **自動でロール追加**(エラーにしない)
- メール送信ロジック:
  - 新規アカウント → 従業員招待メール(72h・PW設定必要)
  - 既存アカウント(管理者) → 「従業員ロールが追加されました」通知メール(PW再設定不要)

#### 16.4.2 STEP 5 管理者登録モーダル(2方式タブ)

| タブ | UI表記 | 内部処理 | 送信メール |
|---|---|---|---|
| 新規登録 | `登録 & 招待送信` | アカウント新規作成+管理者ロール付与 | 管理者招待メール(72h・PW設定必要) |
| 従業員から選択 | `管理者として登録 (昇格通知のみ)` | 既存アカウントに管理者ロール追加のみ | 昇格通知メール(PW再設定不要) |

**重要なバリデーション**:
- 「新規登録」タブでメアド入力時、**従業員台帳と重複チェック**
  - 重複あり → エラー「既に従業員として登録されています。『従業員から選択』タブで管理者に昇格してください」
  - 重複なし → そのまま新規アカウント作成

### 16.5 CSV一括インポート登録での実装

#### 16.5.1 インポートテンプレート拡張

管理者CSVに `mode` 列を追加:

## 20. エンジニア実装確認(2026-05-13 夜): アカウント衝突回避の本番仕様

### 20.1 確認結果

§16「アカウント衝突回避と権限マージ設計」の本番実装について、エンジニアに確認した結果:

> **新規登録で従業員登録と管理者登録を同時にした場合、1アカウントで、後続の管理者登録で権限を上書きする仕様にしたため、コリジョンはない**

### 20.2 §16 提案との対応

| §16 設計案 | エンジニア実装 | 結果 |
|---|---|---|
| 1メアド = 1アカウント | 同上(1アカウント統合) | ✓ 一致 |
| ロール累積(マージ)モデル | 後続登録(管理者)で権限を上書き | ✓ 同等(管理者ロールは従業員機能も包含) |
| メアド衝突時の構造的回避 | 後勝ち上書きで一意性保証 | ✓ コリジョン発生せず |

### 20.3 実装上の挙動(エンジニア仕様)

**同一メアドで複数経路から登録された場合:**
- 設定画面登録(STEP 4 従業員) + 設定画面登録(STEP 5 管理者) → 1アカウント、管理者権限で上書き
- CSV(従業員) + CSV(管理者) → 1アカウント、管理者権限で上書き
- どの経路の組合せでも 1メアド=1アカウント が維持される

**メール送信ロジック:**
- 後続登録で「管理者として上書き」された場合、管理者として招待/通知メールが送信される
- 「従業員招待メール+管理者招待メール」の二重送信は発生しない(1通に集約)

### 20.4 モックでの表現

モック側では引き続き以下の動線で「衝突回避が成立している」前提で UI を提供:
- 設定画面登録モーダル(管理者): 「既存従業員から選択」タブで明示的に昇格動線
- CSVインポート完了画面: 「✉️ 全員に招待メールを一括送信」で適切なメール種別を内訳表示
- 同一メアド衝突の検知・警告は本番 API/DB レイヤで処理(モック範囲外)

### 23.19 管理者登録 4分岐ロジック + 招待差し替え + CSVプレビュー (2026-05-18 さらなる追加)

#### 背景
従業員と管理者の登録動線が独立しているため、メアド重複時の「**二重URL送信**」「**race condition**」「**ユーザー操作順序による状態不整合**」のリスクが顕在化。全動線 (モーダル / バッチUI / CSV) で共通の識別ロジックと整合性ガードが必要。

#### 4分岐識別ロジック (resolveAdminRegistrationType)
メアドを以下の優先順位で判定:
1. **invalid_format** (形式エラー)
2. **already_admin** (既に管理者登録済) → エラースキップ
3. **emp_invited** (招待中従業員) → 招待差し替え (旧無効化 + 新発行)
4. **emp_active** (PW設定済従業員) → 通知メール(既存登録版)
5. **new** (該当なし) → 招待メール(新規版)

各分岐にバッジ表示色 + 送信メール種別 + バックエンド処理を割り当て。

##### ⚠️ 後続実装事項: retired (退職判定)
- 現バックエンドの `employees` テーブルに退職判定可能なカラム (status / retired_at) が**存在しない**ため、本リリースでは退職判定を実装しない。
- 将来 `employees.status` カラム + 退職処理 API + 退職時の管理者権限自動解除フローを別チケットで実装後、5分岐目として再導入予定。
- 暫定対応: 退職者を select に表示してしまう前提で運用、誤登録は管理者一覧の削除ボタンで対応。

#### 新規モック画面 3点
- **page-admin-invite-replacement** … 招待差し替え版メール (件名「【再送・差し替え】」、旧無効化警告、CTA=PW設定)
- **page-invite-invalid-admin** … 旧URL アクセス時のエラー画面 (「この招待は無効です。最新メールをご確認ください」)
- **page-admin-import-preview** … CSVインポート プレビュー (5バッジ表示 + 行別判定 + 手動上書き + サマリブロック + 登録実行)

#### モーダル側の対応
- **emp タブ select** に「⏳ 招待中(要差し替え)」明示 + 既管理者除外 (退職者除外は後続実装)
- **invited 従業員選択時** に警告ダイアログ (招待差し替えの予告)
- **new タブメアド入力**にリアルタイム5分岐判定バッジ (b-setup-admin-overlay + page-d2 両方)

#### バックエンド要件 (エンジニア実装事項)
1. **import_jobs テーブル**: id / type / status (pending/processing/committed/failed) / committed_at / row_count / error_count
2. **「同一メアドの有効招待トークンは常に1つ」** atomic 制約 — 新発行時に旧トークン即失効 (1トランザクション)
3. **旧トークンアクセス時のエラーレスポンス** — 404ではなく専用エラー (画面遷移用)
4. **同一 tenant の排他制御** (同type 1ジョブのみ実行)
5. **ログ/監査**: 招待発行 / 差し替え / 失効 / 権限付与解除

#### 後続実装事項 (別チケット)
1. **employees.status カラム追加** (active / invited / retired)
2. **employees.retired_at カラム追加** (退職日記録)
3. **退職処理 API** (`PATCH /employees/:id/retire`)
4. **退職時の管理者権限自動解除フロー** (もしくは手動解除の運用ルール)

### 23.14 設計原則 (本日の学び)
- **画面と状態は 1対1 対応** (D案思想)
  - 同じ画面が複数 mode を持つと、デモ操作や状態切替時にコンタミが発生
  - 別画面 (別 HTML) として完全分離するのが最も保守性が高い
- **設定画面登録 ⇄ CSV一括インポート は完全独立した動線**
  - データソースも別 (sessionStorage.b0Draft_N vs localStorage.import_in_progress)
  - 同時シナリオは想定外、片方ずつ検証する前提
- **読み取り専用 UI は明示的に**
  - readonly 属性 + グレー背景 + not-allowed カーソル + ラベル明示
  - オーナー指定PW など、ユーザー編集を防ぐべき箇所で活用

### 23.15 整合性ガード追加 (本日後段の修正)
完了画面/管理者一覧パネル/通常TOP のコンタミを構造的に防止する 2 つの整合性ガードを実装。

#### ガード1: b0SetState 末尾の「STEP 5 パネル非表示時 UI 復元」
- **背景**: `b0ShowStep5Panel()` が STEP 5 表示中に `b0-completed-banner.style.display='none'` と `b0-page-title='管理者一覧 (STEP 5・任意)'` を書き換える。STEP 5 パネルを閉じて別画面に遷移し戻ってきた時、書き換えが残るリスク。
- **実装**: `b0SetState` の末尾で `b0-step5-panel` の表示状態を判定し、非表示の場合は必ず `b0-completed-banner.style.display=''` + `b0-page-title.textContent='初期セットアップ'` に復元。
- **効果**: page-b0 に遷移するたびに `b0SetState` が呼ばれるため、STEP 5 パネル非表示時の状態が自動的に正規化される。

#### ガード2: applyB0EmptyMode の「draft / 必須未完検知時 alert 強制」
- **背景**: ユーザーがナビから直接 `page-b0-empty` にアクセスした場合、`b0EmptyMode='normal'` だと STEP 1〜5 中断 draft があってもバナーが表示されない。
- **実装**: `applyB0EmptyMode('normal')` が呼ばれても、`b0LoadDone() < 4` または `b0HasDraft(1..5)` のいずれかを満たす場合は内部で `mode='alert'` に切替えてから処理続行。
- **効果**: 中断状態が残っている限り、TOP にどのルートでアクセスしても必ず再開バナーが表示される。

#### 効果サマリ
- 完了画面 ⇄ 管理者一覧 ⇄ 通常TOP の 3画面でコンタミが構造的に発生しない
- デモボタン操作・ナビ直接遷移・モーダル経由のいずれでも整合
- ユーザー操作パスに依存しないガード設計
