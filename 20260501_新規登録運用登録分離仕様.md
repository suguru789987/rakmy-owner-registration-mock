## 1. 設計コンセプト

タイムレコード利用開始に必要な4項目（業態／店舗／雇用区分／従業員）の登録について、**「新規登録モード」と「運用登録モード」を画面・UIフィードバック・順序強制度で完全分離** します。

### 設計原則

| 観点 | 新規登録モード | 運用登録モード |
|---|---|---|
| **目的** | 利用開始のための4項目必須登録 | 既存運用での追加・編集 |
| **順序** | 必須（① → ② → ③ → ④） | 任意 |
| **強制度** | 強誘導（CTAモーダル） | 軽量示唆（CTAトースト） |
| **画面** | `setup-*` 専用ページ（ウィザード式） | `*-settings` 通常設定画面（一覧+追加） |
| **UI** | クリーンなステップカード | 一覧表 + 「+ 追加」 |

## 3. 画面仕様

### 3.1 B-0 初期セットアップ (`#b0`)

**役割**: 4項目登録の進捗可視化・順序提示・ステップ画面への誘導ハブ

**最上部の構成（共通）**:
- `b0-completed-banner`: done=4 のときに「🎉 初期登録は完了しています」を表示
- 登録方法選択カード `method-picker`: 「設定画面で1件ずつ登録」/「CSVで一括インポート（推奨）」を選択。初期表示は **CSV一括（推奨）**。

#### 3.1.1 manualモード（設定画面で1件ずつ登録）

`ow-card` 内に **4ステップカード（ow-step）** のみを縦並びで表示:
- 1: 業態登録 / 2: 店舗登録 / 3: 雇用区分登録 / 4: 従業員登録
- 各カード: ステップ番号 + タイトル + 説明 + 前提登録 + 「登録済み: N件」 + CTAボタン
- CTAボタン: 「登録する →」（未完了） / 「再編集 →」（完了済み）
- ロック表示: 前ステップ未完了時は `is-locked` + ボタン disabled
- ❗ 進捗リング・パーセント・残数・ステップマーカーは **非表示**（2026-05-11 簡素化）

#### 3.1.2 importモード（CSV一括インポート）（推奨）

`import-panel` 内に以下を縦並びで表示:

1. **方式選択（分岐枠）**: 独立ボックスでピル選択
   - 📋 テンプレートで取り込み（推奨タグ）
   - 🔄 外部勤怠サービスから取り込み
2. **📋 インポートガイド** リスト:
   - テンプレートモード時: **Step 1〜4**（業態 / 店舗 / 雇用区分 / 従業員）
   - 外部連携モード時: **Step 1〜2**（Step 1=業態(テンプレ補完) / Step 2=店舗・雇用区分・従業員(外部一括)）
   - 各行: 番号アイコン + タイトル + 状態バッジ（進行中/✓完了/🔒 Step N完了後）+ CTA
3. **📌 後追加項目（推奨）**: 管理者をテンプレで取り込み（任意・モード共通）
4. **🛠 デモ用シミュレーション**: 静的モック⇄動的モックの連携擬似ボタン

> ⚠️ **B案採用（2026-05-11）**: 「方式選択」は分岐選択（性質: ピル即時切替）でガイドリストとは別枠。ガイドリストは **実取込作業のみ**（性質: 画面遷移＋数分かかる）を Step 1 から番号付け。

**CTAボタン共通仕様**:
- 「インポート画面を開く →」 → 別タブで import-spec `#tpl-setup` / `#saas-setup` に遷移（旧 setup-import-* は §13.2-C で完全撤去）
- 「外部連携画面を開く ↗」 → 別タブで rakmy-timerecord-import-spec/#saas-setup
- 「管理者をテンプレで取込 ↗」 → 別タブで rakmy-timerecord-import-spec/#tpl-setup
- 順序違反は `is-locked` でガード

**完了後（done=4）**:
- 最上部の `b0-completed-banner` のみで通知（インポートパネル内・ウィザード内の重複バナーは廃止）
- サイドバーから「📋 初期セットアップ」項目は非表示

#### 3.4.3 ✅ マスタ参照の連動（2026-06-24 決定・反映済み）

登録済みエンティティ（マスタ）を、それを参照する別フォームのプルダウンに**動的反映**する。あるエンティティを登録/削除すると、参照側プルダウンの選択肢が即時に増減する。

- **表示範囲**: バーは**運用段階（会社・店舗・従業員・雇用区分などを作成した後）の運用・閲覧画面のみ表示**。**認証・登録・オンボーディングでは非表示**（切替対象の会社・店舗が未確定 or 作成中のため）＝ログイン・自己登録(a1〜a4)・パスワードリセット(e1〜e3)・招待メール/PW設定(admin-pw-setup, c4, c5, d3, d4, invite系, mail-formats)・**管理者/従業員の登録・編集フォーム(d2, d1e, c1e-*)**・初期セットアップ(b0)・setupモードの各設定画面。実装は `NO_SCOPE_BAR` セット＋`body.no-scope-bar` でオフセットも復元。

#### F. URLパラメータによる方式・ステップのスキップ（重要）

インポート画面へは **2つの起点で挙動を分ける**:

##### F-1. 初期セットアップからの起点（方式選択スキップ）

b0 の **方式選択（分岐枠）** で方式を選択済みなので、インポート画面では **方式選択画面をスキップして直接該当ステップへ遷移**。

URL: クエリパラメータ付き
```
?method=template&step=businesstype
?method=template&step=store
?method=template&step=emptype
?method=template&step=emp
?method=external&step=store
?method=external&step=emptype
?method=external&step=emp
```

| パラメータ | 値 | 効果 |
|---|---|---|
| `method` | `template` | テンプレート方式の画面で起動 |
| `method` | `external` | 外部連携方式の画面で起動 |
| `step` | `businesstype` | 業態のアップロード画面 |
| `step` | `store` | 店舗のアップロード画面 |
| `step` | `emptype` | 雇用区分のアップロード画面 |
| `step` | `emp` | 従業員のアップロード画面 |

URL例:
```
/import?method=template&step=businesstype  → テンプレート > 業態アップロード画面に直接遷移
/import?method=external&step=store  → 外部連携 > 店舗マッピング画面に直接遷移
```

##### F-2. 運用設定画面（既登録更新）からの起点

> ✅ **改訂（2026-06-03 確定）**: 運用側でも**方式選択モーダルを先に通す動線（§23.20）**に変わったため、**方式を選択済みなら F-1 同様 `?method=X&step=Y` 付きで `/import` に遷移し、インポート画面側の方式選択をスキップ**する。方式未選択のまま直接アクセスした場合のみ方式選択画面を表示。

| 起点の状態 | URL | インポート画面の表示 |
|---|---|---|
| 運用設定画面で**方式選択モーダルを通過（選択済み）** | `?method=X&step=Y` 付き | 方式選択スキップ → 該当ステップ画面 |
| `/import` に**直接アクセス（方式未選択）** | クエリパラメータなし | 方式選択画面（テンプレ/外部）を表示 |

**理由**: 運用側で既に方式選択モーダルを通している場合、インポート画面で再選択を求めるのはUX重複（Q14）。一方、方式未選択の直接アクセスでは柔軟性のため方式選択UIを残す。
- 既存業態の店舗を1件だけ追加 → テンプレート方式
- 別の勤怠サービスから一括移行 → 外部連携方式

##### F-3. 2つの起点の差別化（実装まとめ）

| 起点 | URL | インポート画面の表示 |
|---|---|---|
| **初期セットアップ (b0)** | `?method=X&step=Y` 付き | 方式選択スキップ → 該当ステップ画面 |
| **運用設定画面（方式選択モーダル通過済み）** | `?method=X&step=Y` 付き | 方式選択スキップ → 該当ステップ画面（2026-06-03 改訂） |

**理由**: ガイド側で既に「テンプレート / 外部連携」を選択済みなのに、インポート画面でも再選択を要求するのはUX重複。クエリパラメータでスキップさせて、ガイド順にスムーズにインポートできるようにする。一方、運用時は柔軟性を残すためデフォルトの方式選択UIを維持する。

**現状のモック**: 既存インポートモック (rakmy-timerecord-import-spec) はクエリパラメータ未対応のため、現状は b0 起点でも方式選択画面が表示される。本番実装時にクエリパラメータ対応すること。

### 12.6 KoT (King of Time) データソース仕様

KOTのエクスポート機能から取得できる4種類のデータをタイムレコのエンティティにマッピングする。s-saas1 の「KoT キングオブタイム」選択時に以下の5チェック項目を表示。

| # | KoTデータソース | 必須/任意 | 初期チェック | タイムレコ側マッピング先 | 備考 |
|---|---|---|---|---|---|
| 1 | **従業員データ** | **必須** | ✓ ON | 従業員 | 従業員コード/所属コード/姓/名/雇用区分コード/メール/日の契約労働時間/週の契約労働日数 |
| 2 | **従業員一覧** | 任意 | OFF | (給与データ補完) | 給与・契約種別を含む補助データ |
| 3 | **所属グループデータ** | 任意 | OFF | **店舗情報** | 所属コード/所属名/所属グループコード/グループ名 |
| 4 | **単価データ** | 任意 | OFF | **雇用区分** | 所属コード/従業員コード/表示名/基本単価/日給or時給 |
| 5 | **管理者データ** | 任意 | OFF | 管理者 | 管理者コード/表示名 |

#### 設計判断
- **必須は「従業員データ」のみ**: KOTの基幹データを最優先で取り込み
- **店舗・雇用区分・管理者**: ユーザーが必要に応じて選択
- **業態**: KOTに対応データなし → 「②取り込みデータ」の業態テンプレで補完
- **マッピング先表示**: 各任意項目に「→ タイムレコの○○にマッピング」を補足表示（従業員データは既存挙動のため補足なし）

#### data-map 属性とマッピングパネル
| データソース | data-map | マッピング設定パネル |
|---|---|---|
| 従業員データ | `map-kot-emp` | (既存) |
| 従業員一覧 | `map-kot-hr` | (既存) |
| 所属グループデータ | `map-kot-group` | (本番実装で要追加) |
| 単価データ | `map-kot-rate` | (本番実装で要追加) |
| 管理者データ | `map-kot-admin` | (本番実装で要追加) |

#### 本番実装での留意点
- KOTの実際のCSVファイル名規則（例: `employee_info_YYYYMMDD.csv` / `rate_YYYYMMDD.csv` 等）の確認が必要
- 各マッピング設定パネルでKoT列 → タイムレコ列の対応を定義
- 「単価データ」は雇用区分カラム（給与体系・金額・月次契約労働時間・月みなし残業時間）にマップ（旧「時給単位」表記は廃止。§3.4 統一スキーマに準拠）
- 「所属グループデータ」は店舗カラム（店舗名・所属業態）にマップ
- 「管理者データ」は管理者カラム（名前・メールアドレス・担当店舗・権限）にマップ。なお、KOTから取得できる管理者データには **メールアドレスが含まれない** 可能性が高いので、別途入力フォームまたはマージ処理が必要

## 16. アカウント衝突回避と権限マージ設計（2026-05-13 追加）

### 16.1 問題提起

複数の登録経路(設定画面登録/CSV一括インポート)が存在し、それぞれが「従業員」「管理者」を独立して登録できる仕様だと、**同一メアドに複数の招待メールが届くコリジョン**が発生する。

具体的なシナリオ:
1. 従業員CSVインポートで A さん登録 → 従業員招待メール送信
2. 管理者CSVインポート(新規)で同じ A さん登録 → 管理者招待メール送信
3. → A さんに**2通の招待メール**が届き、PW設定URLが2つ存在しユーザー混乱

### 16.2 採用する設計モデル

「**アカウント = メアド単位、ロールは累積**」モデルを採用する。

```
[アカウントテーブル]
- account_id (UUID, 主キー)
- email (UNIQUE)         ← この一意キーで衝突回避
- password_hash
- status (招待中 / 有効 / 期限切れ)
- created_at

[ロールテーブル]
- account_id (FK)
- role_type (employee / admin)
- stores (担当店舗、複数可)
- emptype (雇用区分・employeeロールのみ)
- granted_at
```

**主要原則**:
- 1メアド = 1アカウント (UNIQUE 制約で物理的に保証)
- 同一アカウントが「従業員ロール」と「管理者ロール」を**累積**して持てる
- 招待メールは「アカウント単位」で1通のみ、ロールは「メール本文と権限付与ロジック」で表現

### 16.3 ケース別の挙動(4パターン)

| # | 状況 | システム処理 | 送信メール |
|---|---|---|---|
| **1** | 同時インポート時に同一メアドが両CSVに存在(PW未設定) | 1アカウント作成 + 両ロール累積付与 | **統合招待メール 1通**「従業員+管理者として招待」 |
| **2** | 既に従業員アカウントあり → 管理者ロール追加 | 既存アカウントに管理者ロール追加 | **権限拡張通知 1通**(PW設定不要、既存PWで全ロール利用可) |
| **3** | 既に管理者アカウントあり → 従業員ロール追加 | 同上(ロール追加方向のみ異なる) | 同上(権限拡張通知) |
| **4** | 同一CSV内でメアド重複 | バリデーションエラー(インポート拒否) | (送信なし)CSV修正要求 |

### 16.4 設定画面登録での実装

#### 16.4.1 STEP 4 従業員登録モーダル

- 「登録 & 招待送信 (N件)」ボタンで一括登録+招待メール送信
- 内部処理: 各メアドに対して「アカウント新規作成 OR 既存アカウントに従業員ロール追加」をAPIが判定
- 既存管理者アカウントが存在する場合は **自動でロール追加**(エラーにしない)
- メール送信ロジック:
  - 新規アカウント → 従業員招待メール(72h・PW設定必要)
  - 既存アカウント(管理者) → 「従業員ロールが追加されました」通知メール(PW再設定不要)

#### 16.4.2 STEP 5 管理者登録モーダル(2方式タブ)

| タブ | UI表記 | 内部処理 | 送信メール |
|---|---|---|---|
| 新規登録 | `登録 & 招待送信` | アカウント新規作成+管理者ロール付与 | 管理者招待メール(72h・PW設定必要) |
| 従業員から選択 | `管理者として登録 (昇格通知のみ)` | 既存アカウントに管理者ロール追加のみ | 昇格通知メール(PW再設定不要) |

**重要なバリデーション**:
- 「新規登録」タブでメアド入力時、**従業員台帳と重複チェック**
  - 重複あり → エラー「既に従業員として登録されています。『従業員から選択』タブで管理者に昇格してください」
  - 重複なし → そのまま新規アカウント作成

### 16.5 CSV一括インポート登録での実装

#### 16.5.1 インポートテンプレート拡張

管理者CSVに `mode` 列を追加:
- `new`: 新規入力(アカウント新規作成)
- `promote`: 既存従業員から昇格(既存アカウントに管理者ロール追加)

#### 16.5.2 プレビュー画面での衝突検知

インポート確定前に、システムが以下をチェック:

```
インポート対象: 50件
├─ ✓ 新規アカウント作成: 42件 → 招待メール42通予定
├─ 🔄 既存アカウントへロール追加: 6件 → 権限拡張通知6通予定
└─ ⚠️ 同一CSV内重複でエラー: 2件 → 修正してください
```

#### 16.5.3 インポート完了画面での一括送信ボタン

`import-spec` の完了画面(`s-tpl3`/`s-saas4`)に**「✉️ 招待メールを一括送信」CTA**を配置。

ボタン押下時の動作:
- 新規アカウント分: 招待メール(72h)を送信
- ロール追加分(既存アカウント): 権限拡張通知を送信
- 同一メアドが従業員/管理者双方に存在する場合は **1通に統合**

### 16.6 メール種別の整理

| メール種別 | 送信タイミング | PW設定要否 | 内容 |

| **従業員招待** | 従業員ロール付き新規アカウント作成時 | 必要(72h有効) | 従業員としての挨拶 + PW設定リンク |

## 20. エンジニア実装確認(2026-05-13 夜): アカウント衝突回避の本番仕様

### 20.1 確認結果

§16「アカウント衝突回避と権限マージ設計」の本番実装について、エンジニアに確認した結果:

> **新規登録で従業員登録と管理者登録を同時にした場合、1アカウントで、後続の管理者登録で権限を上書きする仕様にしたため、コリジョンはない**

### 20.2 §16 提案との対応

| §16 設計案 | エンジニア実装 | 結果 |
|---|---|---|
| 1メアド = 1アカウント | 同上(1アカウント統合) | ✓ 一致 |

| メアド衝突時の構造的回避 | 後勝ち上書きで一意性保証 | ✓ コリジョン発生せず |

### 20.3 実装上の挙動(エンジニア仕様)

**同一メアドで複数経路から登録された場合:**
- 設定画面登録(STEP 4 従業員) + 設定画面登録(STEP 5 管理者) → 1アカウント、管理者権限で上書き
- CSV(従業員) + CSV(管理者) → 1アカウント、管理者権限で上書き
- どの経路の組合せでも 1メアド=1アカウント が維持される

**メール送信ロジック:**
- 後続登録で「管理者として上書き」された場合、管理者として招待/通知メールが送信される
- 「従業員招待メール+管理者招待メール」の二重送信は発生しない(1通に集約)

### 23.20.7 エラー対応 UI: 均一 3 ボタン

**設計原則**: エラー種別判別による推奨アクション切替 ❌ → **3 ボタン均一 + データ項目別の挙動** ⭕

理由:
- エラー判別が不可な場合に推奨が出せず漏れる
- 判別ロジックの精度に運用が依存しない
- 「3 ボタンの意味を 1 度覚えれば全エラー行で同じ操作」

| ボタン | 用途 | データ項目別の挙動 |
|---|---|---|
| ➕ マスタに追加 | 新しい値を該当マスタへ登録 | businesstype 行 → `openMasterAddRedirect('businesstype',value)` / emptype 行 → `openMasterAddRedirect('emptype',value)` ... |
| ✏️ 行編集 | CSV 側の値を直接書き換え | `openRowEditModal(field, current, suggested)` |
| ⊘ スキップ | この行のみ取込対象から除外 (`markRowSkipped`) / 元に戻す可 |

**+ 行レベル / バナーレベル**:
- 行レベル「➕ マスタに追加」 → **単一行モーダル** (`#master-add-overlay`、縦並びフォーム)
- バナー「➕ 未登録 N件をまとめて登録」 → **複数行テーブルモーダル** (`#bulk-master-add-overlay`、テーブル形式 + 行追加可)

#### 動線判別ロジック (routing)

import-spec 側 `initFromUrl()` (`?type` / `?mode` / `?goto` / `?openSavedMappingList` を解釈):

- `?type=X&mode=tpl` → 動線 ② `s-tpl-single-X` に直行
- `?type=X&mode=saas` → 動線 ② `s-saas-single-X` に直行
- `?type=X&mode=saas&goto=mapping` → 動線 ② マッピング画面に直行 (one-shot、`goto` は適用後 URL から削除)
- URL hash `#tpl-setup` or `#saas-setup` → 動線 ③ セットアップモード
- URL パラメータ無し → 動線 ① s-top → ユーザがサイドバーで選択

#### コンタミ防止ルール

- **動線 ② アップ画面では setup 用 UI を出さない** (順序必須バッジ、TOP 連動再開ボタン、一時保存 UI など)。`s-tpl-single-*` / `s-saas-single-*` は専用画面なので物理的に混入不可。
- **動線 ① / ③ で同じ s-tpl1 を共有しているが UI モードは `applyTplMode('ops'|'setup')` で切替**。hash 検知で判定。
- **下流画面 (確認・プレビュー・完了)** は動線 ① と ② で共通 (`*-single`)、動線 ③ は別 (`s-tpl2` / `s-saas4`)。
- **新画面・新動線追加時は本表を必ず更新**、テスト時は 3 動線 × 5 種類 × 2 モード = 最大 30 ケースを通す。

#### 動線間遷移可否 (横断)

- 動線 ② アップ画面のヘッダー右上「📄 テンプレ取込に切替 / 🔄 外部勤怠取込に切替」リンクで **mode のみ切替** 可 (同じ key を維持、`singleTplSwitchToSaas` / `singleSaasSwitchToTpl`)。
- 動線 ② → owner-mock の「📥 CSV取込設定」マッピング一覧 (`?openSavedMappingList=key&src=src` deep-link)。一覧の戻るボタンで動線 ② のマッピング画面へ復帰 (`?goto=mapping`)。
- 動線 ① ⇄ 動線 ② の直接遷移は非対応 (起点が違うため別タブで開く想定)。
- 動線 ③ → 動線 ② / ① の直接遷移は禁止 (初期セットアップ完了後に各動線へ開放される設計)。

#### 判断軸の総括

PdM が一貫して優先した判断軸:

1. **判別不可ケースで UI が破綻しない設計** (均一 3 ボタン、進捗バー必須経路のみ等)
2. **責務境界の明確化** (CSV取込設定 ≠ 外部連携、マッピング ≠ アクションフロー)
3. **冗長を削る** (バナー圧縮、状況詳細統合、不要トグル削除)
4. **ブラウザネイティブ UI を排除** (alert → トースト/モーダルに統一)
5. **明示的なユーザーアクション** (自動遷移より「次へ」明示クリック)
6. **状態管理の排他制御** (checkbox → radio、ブラウザ強制を活用)
7. **AI が出した「自分の構造に従いがち」な提案を見抜く** メタ判断 (推奨ボタン → 均一 rollback 等)

これらは Claude が **「網羅性 + 構造化 + ロジック完備」を優先する傾向** とは別の判断軸で、PdM の固有価値が出る領域。

### 27.2 確立した判断軸（再利用可能な原則）

1. **データスキーマは CSVインポートテンプレを正**（登録/編集/import の3経路で同一項目を捕捉、経路欠落を作らない）。
2. **矛盾時の優先は「コード(index.html) > 仕様書」**。モックが de-facto 最新のため、仕様書は後追いで現行化する。
3. **ドキュメントは「現在の正」と「変更履歴」を物理分離し、入口を1つ設ける**（前半章＋§2.1.1/§24〜§26 が現在の正、§12〜§23 は履歴）。新たな対応表を増やすより既存の正に集約してドリフト源を作らない。
4. **設計転換は前半章・README・データ・他リポへ逆伝播**してから完了とする（前半凍結が今回の大量の問い直しの根因）。
5. **コピー/文面はCS領域として分離**、PdM/エンジニアは種別・条件・要否までを確定する。

### 27.3 今回の根本学習

問い直し19件の主因は、**仕様書前半（§2〜§8）が初期設計のまま凍結され、後続の設計転換（setup-*廃止／*-ops-settings分離／名称統一／5テンプレ化）が逆伝播されなかったドリフト**だった。AI補助の高速プロトタイピングでモックが先行し、ドキュメント・READMEが取り残された結果である。対策として「現在の正」を集約（§2.1.1/§26）し、矛盾時の正の所在を明文化した。
