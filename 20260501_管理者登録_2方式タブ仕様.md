## 1. 変更サマリ

これまで管理者登録は「新規入力のみ」だったが、**従業員台帳に既に登録済みの従業員を管理者として昇格させる導線**を追加する。
モーダル内のスライドタブで2方式を切替する。

| 方式 | 用途 | 入力 |
|---|---|---|
| **新規登録** | 従業員台帳に未登録の人を直接管理者として追加 | 名前 / メール / 担当店舗（手入力） |
| **従業員から選択** | 既存従業員を管理者に昇格 | 従業員プルダウン（メール・雇用区分・所属店舗は台帳から自動表示） / 担当店舗（所属店舗を初期チェック） |

## 3. UI仕様

### 3.1 モーダル構造

```
┌──────────────────────────────────────┐
│ 管理者を登録                       ✕ │  ← ヘッダー
├──────────────────────────────────────┤
│ ┌─────────────┬─────────────┐        │
│ │ 新規登録 ●  │ 従業員から   │        │  ← スライドタブ
│ │             │   選択       │        │     （タイトル直下）
│ └─────────────┴─────────────┘        │
│ ─────────────────────────────────    │
│ [選択中タブのフォーム]                  │
│ ─────────────────────────────────    │
│ 💡 PWリセット案内                      │
│                  [キャンセル][登録]    │
└──────────────────────────────────────┘
```

### 3.2 タブ「新規登録」フォーム（従来仕様維持）

| 項目 | 必須 | 型 | 備考 |
|---|---|---|---|
| 名前 | ✳ | text | 例: 鈴木 花子 |
| メールアドレス | ✳ | email | 例: suzuki@example.jp |
| 担当店舗 | ✳ | checkbox（複数） | 渋谷店 / 新宿店 / 池袋店 / 北海道支店 |

**インフォメーション表示**:
> 💡 従業員台帳には登録されていない方を、管理者として直接追加します。登録後、初期パスワードはオーナーが管理者へ口頭/チャット等で共有してください。

### 3.3 タブ「従業員から選択」フォーム（新規）

| 項目 | 必須 | 型 | 備考 |
|---|---|---|---|
| 従業員を選択 | ✳ | select | 従業員台帳より取得（雇用区分 / 所属店舗を併記表示） |
| 選択サマリ | - | 表示のみ | 名前 / 雇用区分 / 所属店舗 |
| メールアドレス | - | text(readonly) | 従業員台帳より自動入力 |
| 担当店舗 | ✳ | checkbox（複数） | **所属店舗を初期チェック**、必要に応じ追加可 |

**インフォメーション表示**:
> 💡 既に従業員台帳に登録済みの方を、管理者として追加します。氏名を選択するとメールアドレスが自動入力されます（従業員台帳の値を参照）。

## 4. インタラクション仕様

### 4.1 タブ切替

- タイトル直下のセグメント型タブ（左:新規登録 / 右:従業員から選択）
- 選択中タブ: 背景白＋緑文字（`#1a9e7e`）＋微シャドウ
- 非選択タブ: グレー文字（`#888`）
- 切替時、もう一方のパネルの入力値はクリアしない（タブ往復で値が保持される想定）
- モーダルを閉じる/開き直しで初期タブ「新規登録」にリセット

### 4.2 従業員選択時の自動連携

```
従業員プルダウン onChange
  ├─ 未選択（空value）の場合
  │   └─ サマリ非表示 / メールクリア / 担当店舗チェック全解除
  └─ 選択ありの場合
      ├─ サマリ表示（名前 / 雇用区分 / 所属店舗）
      ├─ メール自動入力（readonly）
      └─ 担当店舗チェックボックス: 所属店舗のみ初期チェック
```

### 4.3 ボタン挙動

| ボタン | 挙動（実装側で要確認） |
|---|---|
| `登録` | フォームバリデーション後、管理者一覧（D-1）へ戻る |
| `キャンセル` | 入力破棄、D-1へ戻る |
| `✕` | キャンセルと同等 |

## 7. エンジニア確認事項（技術側で要判断）

仕様/UXは本書および本モックで確定済み。エンジニア側で判断が必要な技術項目のみ以下に記載。

### 7.1 API
1. **従業員一覧API**: 既存の `GET /api/employees` 等を「従業員から選択」プルダウンに流用可能か。レスポンス項目に雇用区分・所属店舗・メールアドレスが含まれるか（不足する場合の拡張方針）
2. **登録API**: 「新規登録」と「従業員から選択」で同一エンドポイント（`POST /api/admins` でboolフラグ判別）か／別エンドポイント（`POST /api/admins/from-employee` を分離）か。実装/保守容易性で判断

### 7.2 DB構造
3. **管理者の表現**: 「従業員から選択」での管理者登録は、`users.role` を `manager` に変更する方式（単一テーブル / ロールカラム）か／別 `administrators` テーブルに紐付けレコードを作る方式（多対多relation）か。既存スキーマとの整合で判断

### 7.3 メール送信（D-3 / D-4 共通）
4. **送信タイミング**: 登録APIのレスポンス内で同期送信か／Sidekiq等のキュー経由で非同期送信か
5. **送信失敗時のリトライ**: あり/なし、上限回数、失敗時のオーナーへの通知方法

---

> **仕様確定済み事項（参考）** — 以下はPdM/UX側で確定済み、エンジニア確認不要
> - 重複防止: 管理者登録済みの従業員はプルダウンから除外（モック側でフィルタ前提）
> - 権限制限なし: 雇用区分問わず管理者登録可
> - 招待中従業員も管理者登録可
> - 所属外店舗も担当店舗に追加可（同一会社内）
> - 「従業員から選択」時のメール: readonly固定（従業員台帳の値を参照）
> - タブ切替時の入力値: 保持（往復しても消えない）
> - 所属店舗の初期チェック: ON
> - バリデーションエラー: 既存 `.al-w` 警告アラートパターン流用
> - D-1E 編集モーダル: タブなしで継続
> - 削除時挙動: 管理者削除しても従業員レコードは残す
