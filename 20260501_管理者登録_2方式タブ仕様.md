## 1. 変更サマリ

これまで管理者登録は「新規入力のみ」だったが、**従業員台帳に既に登録済みの従業員を管理者として昇格させる導線**を追加する。
モーダル内のスライドタブで2方式を切替する。

| 方式 | 用途 | 入力 |
|---|---|---|
| **新規登録** | 従業員台帳に未登録の人を直接管理者として追加 | 名前 / メール / 担当店舗（手入力） |
| **従業員から選択** | 既存従業員を管理者に昇格 | 従業員プルダウン（メール・雇用区分・所属店舗は台帳から自動表示） / 担当店舗（所属店舗を初期チェック） |

## 3. UI仕様

### 3.1 モーダル構造

```
┌──────────────────────────────────────┐
│ 管理者を登録                       ✕ │  ← ヘッダー
├──────────────────────────────────────┤
│ ┌─────────────┬─────────────┐        │
│ │ 新規登録 ●  │ 従業員から   │        │  ← スライドタブ
│ │             │   選択       │        │     （タイトル直下）
│ └─────────────┴─────────────┘        │
│ ─────────────────────────────────    │
│ [選択中タブのフォーム]                  │
│ ─────────────────────────────────    │
│ 💡 PWリセット案内                      │
│                  [キャンセル][登録]    │
└──────────────────────────────────────┘
```

### 3.2 タブ「新規登録」フォーム（従来仕様維持）

| 項目 | 必須 | 型 | 備考 |
|---|---|---|---|
| 名前 | ✳ | text | 例: 鈴木 花子 |
| メールアドレス | ✳ | email | 例: suzuki@example.jp |
| 担当店舗 | ✳ | checkbox（複数） | 渋谷店 / 新宿店 / 池袋店 / 北海道支店 |

**インフォメーション表示**:
> 💡 従業員台帳には登録されていない方を、管理者として直接追加します。登録後、初期パスワードはオーナーが管理者へ口頭/チャット等で共有してください。

### 3.3 タブ「従業員から選択」フォーム（新規）

| 項目 | 必須 | 型 | 備考 |
|---|---|---|---|
| 従業員を選択 | ✳ | select | 従業員台帳より取得（雇用区分 / 所属店舗を併記表示） |
| 選択サマリ | - | 表示のみ | 名前 / 雇用区分 / 所属店舗 |
| メールアドレス | - | text(readonly) | 従業員台帳より自動入力 |
| 担当店舗 | ✳ | checkbox（複数） | **所属店舗を初期チェック**、必要に応じ追加可 |

**インフォメーション表示**:
> 💡 既に従業員台帳に登録済みの方を、管理者として追加します。氏名を選択するとメールアドレスが自動入力されます（従業員台帳の値を参照）。

## 4. インタラクション仕様

### 4.1 タブ切替

- タイトル直下のセグメント型タブ（左:新規登録 / 右:従業員から選択）
- 選択中タブ: 背景白＋緑文字（`#1a9e7e`）＋微シャドウ
- 非選択タブ: グレー文字（`#888`）
- 切替時、もう一方のパネルの入力値はクリアしない（タブ往復で値が保持される想定）
- モーダルを閉じる/開き直しで初期タブ「新規登録」にリセット

### 4.2 従業員選択時の自動連携

```
従業員プルダウン onChange
  ├─ 未選択（空value）の場合
  │   └─ サマリ非表示 / メールクリア / 担当店舗チェック全解除
  └─ 選択ありの場合
      ├─ サマリ表示（名前 / 雇用区分 / 所属店舗）
      ├─ メール自動入力（readonly）
      └─ 担当店舗チェックボックス: 所属店舗のみ初期チェック
```

### 4.3 ボタン挙動

| ボタン | 挙動（実装側で要確認） |
|---|---|
| `登録` | フォームバリデーション後、管理者一覧（D-1）へ戻る |
| `キャンセル` | 入力破棄、D-1へ戻る |
| `✕` | キャンセルと同等 |

## 7. 実装にあたり確認事項（要回答）

エンジニア実装前に以下を仕様確定したい：

### 7.1 ビジネスロジック
1. **重複防止**: 既に管理者登録済みの従業員を「従業員から選択」プルダウンから除外するか／選択時にエラー表示するか
2. **権限制限**: 雇用区分が `バイトさん` の従業員を管理者として登録できるか（業界慣習的に管理者≒社員以上が想定される）
3. **招待中ステータスの従業員**: 従業員側がまだ招待中（メール認証未完了）の場合、管理者登録を許可するか
4. **担当店舗の上限**: 所属店舗以外も担当店舗に追加可能とした場合、どこまで許容するか（全店舗OK / 同一会社内のみ等）
5. **メールアドレス**: 「従業員から選択」時のメールは編集不可（readonly）でよいか／編集可とする場合、従業員台帳のメールも同期更新するか

### 7.2 データ・API
6. **従業員一覧API**: 既存の `GET /api/employees` 等の利用可否、レスポンス項目に雇用区分・所属店舗が含まれるか
7. **登録API**: 「新規登録」と「従業員から選択」で同一エンドポイントか／別エンドポイント（`POST /api/admins` vs `POST /api/admins/from-employee`）か
8. **トランザクション**: 「従業員から選択」での管理者登録は、従業員レコードに `is_admin=true` フラグを立てる方式か／管理者テーブルに別レコードを作る方式か

### 7.3 UX
9. **タブ切替時の入力保持**: もう一方のタブに切替→戻った時に入力値を保持するか／クリアするか
10. **所属店舗の自動チェック**: 「従業員から選択」で所属店舗を初期チェックする挙動は妥当か（外したい場合のUIは現状チェック解除のみ）
11. **バリデーションエラー**: モーダル内の表示位置・スタイル（既存の `.al-w` 警告アラート利用想定）

### 7.4 既存挙動の整合
12. **D-1E（管理者編集）**: 編集モーダルにはタブを設けない方針で確定してよいか（既存従業員と紐付け済みかどうかは編集UIに反映しない、で問題ないか）
13. **削除挙動**: 「従業員から選択」で登録した管理者を削除した場合、従業員レコード自体は残す前提でよいか
